Every variable you've written so far lives on the stack &mdash; born when its scope opens, gone the instant it closes. That's fine until you don't know how much memory you'll need until the program is already running: a file of unknown size, a list that grows as the user types. For that, C gives you the <b>heap</b> &mdash; a separate pool of memory you control entirely by hand.

`malloc` asks the heap for a block of bytes and hands you back a pointer to it; `free` gives that block back. Get the pairing right and you have memory that outlives any single function. Get it wrong &mdash; and in the next lesson, on purpose, we will &mdash; and things get ugly.

## Stack vs heap, one more time

Flip back to Concept 2: a local variable lives on the stack, and its space is reclaimed the instant the function that owns it returns &mdash; automatically, with zero work from you. That's fast and safe, but it only works because the compiler knows the size and the lifetime in advance.

The heap makes the opposite trade. You explicitly ask for a block, and it stays reserved &mdash; even after the function that requested it returns &mdash; until you explicitly give it back. Nothing does that for you automatically. That's the entire reason `malloc` and `free` exist.

## Requesting memory with malloc

`malloc(n)` asks the heap for n bytes and returns a pointer to the first one &mdash; or `NULL` if it couldn't find them. Multiplying by `sizeof`, as in `malloc(5 * sizeof(int))`, keeps the byte count matched to whatever type you actually want: five ints, not five bytes.

That pointer behaves like any other pointer the moment you have it: index it, dereference it, pass it to functions. The only real difference from a stack variable is who's responsible for cleaning it up &mdash; and this time, that's you.

## Every malloc needs a free

`malloc` never cleans up after itself. If you lose the only pointer to a block &mdash; by overwriting the variable that held it, or returning from a function without freeing it &mdash; that memory is unreachable but still marked in use for as long as your program runs. That's a <b>memory leak</b>.

The rule of thumb: every successful `malloc` should be matched by exactly one `free`, once you're truly finished with the block. Free it too early, though, and you've set up the exact mistake the next lesson is entirely built around.

## Why this matters

You now have every ingredient for the final concept in this level: what happens when you use a pointer after freeing what it pointed to, and why the compiler doesn't stop you. Structs, the stack, the heap, and now allocation all collide in undefined behaviour.
