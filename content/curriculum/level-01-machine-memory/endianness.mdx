You now know that a multi-byte value — a 4-byte int, say — doesn't live at one address, it spans several consecutive ones (back in Lesson 3's memory-strip diagram). That raises a question you've been able to avoid until now: given four bytes and four addresses, which byte goes <i>where</i>?

The answer is: it depends on the machine, and both conventions are alive and well in the real world. This is <b>endianness</b> — a small idea that causes outsized confusion the first time you stare at a hex dump or a network packet and the bytes seem to be "backwards."

## One value, several bytes

Take the 4-byte value `0x12345678`. It needs four consecutive addresses to store it — say 0x100, 0x101, 0x102, and 0x103. The value has a most-significant byte (`0x12`) and a least-significant byte (`0x78`) — and nothing about "four addresses in a row" tells you, by itself, which byte belongs at the lowest address.

That's a genuine design decision every CPU architecture has to make, and different architectures have made it differently. It's entirely internal and invisible as long as one machine both writes and reads its own memory — the trouble starts the moment bytes cross a boundary: to a different machine, across a network, or straight into a debugger showing you raw hex.

## Little-endian vs big-endian

<b>Little-endian</b> stores the least-significant byte at the lowest address — so 0x12345678 is laid out in memory as `78 56 34 12` reading from address 0x100 upward. <b>Big-endian</b> stores the most-significant byte first — `12 34 56 78` — which reads the way you'd naturally write the number on paper.

x86 and x86-64, the family of chips in most laptops and desktops, are little-endian; so is ARM by default (though ARM can actually run either way). Network protocols went the other direction: TCP/IP standardizes on big-endian, which is why it's traditionally called <b>network byte order</b>. Neither convention is "more correct" — they're just two consistent answers to the same design question.

## Why you'll actually run into this

Endianness stops being trivia the moment you look at raw bytes directly: a hex dump of memory, a byte-for-byte view of a network packet, or the contents of a file format. Misjudge the byte order and a value that should read as 1 can appear to read as 16,777,216 — same bytes, wrong assumption about their order.

This is exactly why network programming has dedicated conversion functions — `htons`/`ntohl` and friends — whose entire job is converting between your machine's native byte order and network byte order. You'll meet them for real once you start writing networked C code.
