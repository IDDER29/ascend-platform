Your program can't just reach into the kernel and read a disk block, or write to the network, or spawn a process — the kernel guards all of that. Every single thing your program does that touches the outside world eventually goes through one narrow, well-defined gate: the <b>system call</b>. Even `printf()` — which feels like plain C — is a thin wrapper that, sooner or later, calls `write()`.

Understanding this gate is what turns "C programming" into "systems programming." It's also exactly what you'll be doing constantly inside your shell: every `fork()`, `execve()`, `pipe()` and `dup2()` you're about to learn is a system call.

## User mode vs. kernel mode

The CPU itself enforces a split between <b>user mode</b>, where your program's instructions run with restricted privileges, and <b>kernel mode</b>, where the OS runs with full access to hardware and memory. Your program physically cannot execute a "read this disk sector" instruction in user mode — the hardware would refuse it.

A system call is the one controlled doorway between the two: your code executes a special trap instruction, the CPU switches to kernel mode, the kernel does the privileged work on your behalf, and control returns to user mode with a result. This is why syscalls are slower than ordinary function calls — you're paying for two mode switches, not just a jump.

## Anatomy of a syscall: write()

`write(fd, buf, count)` is about as raw as I/O gets: write `count` bytes from `buf` to whatever `fd` refers to. It returns the number of bytes actually written, or `-1` on failure — and on failure, it never tells you <i>why</i> directly. That's what the global `errno` is for.

This return-value-plus-errno pattern is universal across almost every POSIX syscall, so getting comfortable with it here pays off for the rest of this level — fork(), pipe(), and dup2() all fail the same way.

## The wrapper is a convenience, not the truth

Libc functions like `printf()`, `malloc()`, and `fopen()` are conveniences built on top of a much smaller set of raw syscalls — `write()`, `brk()`/`mmap()`, and `openat()`, respectively. The tool `strace` lets you watch this curtain get pulled back — run any program under it and you'll see the literal syscalls it made, in order, with their real arguments and return values.

## Syscalls aren't free

Every mode switch costs real time — saving registers, changing privilege level, and switching back. That's precisely why `printf()` doesn't call `write()` once per character: it buffers output in user-space memory and only calls `write()` once the buffer fills up (or hits a newline, on a terminal).

Keep this in your back pocket — it explains a subtle bug you'll hit in the shell project: mixing buffered stdio with raw file descriptor tricks like `dup2()` can print things in the "wrong" order if you forget to flush.
