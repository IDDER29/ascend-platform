You now have every ingredient: bits (lesson one), numbers built from them (lesson two), logic to combine them (lesson three), and physical gates that carry out that logic (lesson four). This final lesson of Level 00 assembles all four into the thing that turns a pile of gates into an actual computer: the <b>fetch-decode-execute cycle</b> — the loop every CPU on earth runs, over and over, to run any program at all.

By the end you'll be able to name the stages of that cycle in order, explain what the program counter and instruction register actually do, trace a real instruction through the whole loop, and understand why "running a program" is, physically, nothing more than this loop repeating billions of times a second.

## Instructions are bits too

Here's the twist that ties everything together: instructions — the individual steps of a program — are stored in memory as binary patterns, exactly like the numbers from lesson two. A pattern like `00000011` might mean "add", decided purely by convention, the same way a byte could mean the letter A or the number 65.

The CPU keeps track of where it is with a special register called the <b>program counter (PC)</b>: it always holds the memory address of the next instruction to run. A second register, the <b>instruction register (IR)</b>, holds the instruction currently being decoded and executed, once it's been fetched from that address.

## Fetch, decode, execute

The cycle has three core stages, repeated forever. <b>Fetch:</b> the CPU reads the instruction bits at the address the PC points to, loads them into the IR, then advances the PC to the next address. <b>Decode:</b> a piece of hardware called the control unit examines the instruction's bits and figures out which operation they mean and which gates and circuits need to switch on to perform it. <b>Execute:</b> the actual operation happens — for something like "add", the ALU (arithmetic logic unit, built from exactly the AND/XOR-style gates of the last lesson) computes the result, which then gets written back to a register or memory.

Take a concrete instruction like `ADD R1, R2`: fetch pulls its bits into the IR and moves the PC forward; decode recognizes the opcode as "add" and identifies R1 and R2 as the operands; execute drives those two register values into the ALU's adder circuit and stores the sum. A clock signal keeps every stage, across every part of the chip, in lockstep.

## Why this loop is "a computer"

A CPU doesn't "understand" a program the way a person does — it is simply repeating this same three-stage loop, unimaginably fast (a 3 GHz processor completes roughly three billion cycles every second), always fetching whatever instruction the PC currently points to. Branching, loops, and function calls in a program are, at this level, nothing mystical: they're just instructions that change the PC to something other than "the next address", redirecting the loop's next fetch.

That closes the whole arc of Level 00: a bit's two voltage states, stacked into binary numbers, combined by boolean rules, implemented as physical gates, assembled into a loop that fetches and executes those gate-built instructions forever. That entire stack, bottom to top, is what a computer <i>is</i>. Every level from here — assembly, C, operating systems — is just building further on top of this exact loop.
