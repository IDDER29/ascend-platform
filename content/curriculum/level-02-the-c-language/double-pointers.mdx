You now know two things cold: a pointer holds an address, and pointer arithmetic moves through memory one whole element at a time. Here's the twist that ties them together — a pointer variable is itself just a variable, which means it lives at its own address, which means something else can point <i>at it</i>.

That's all a double pointer is: an `int **` holds the address of an `int *`, which holds the address of an `int`. Two hops instead of one. It sounds like an abstract escalation, but it solves a concrete problem — letting a function change which address <i>you</i> are holding, not just what's stored at that address.

## One more hop of indirection

Take the familiar setup: `int x = 42; int *p = &x;` — `p` holds the address of `x`. Now add one more line: `int **pp = &p;`. Just like `&x` gave you the address of `x`, `&p` gives you the address of `p` — and `pp` stores that. Nothing new is happening except that a pointer is a variable too, and every variable has an address.

To get back to `x` from `pp`, you dereference twice: `*pp` gives you `p`'s value (which is `x`'s address), and `**pp` follows that address to reach `x` itself. Each `*` is one hop back down the chain.

## Two dereferences, one assignment

Run the lab below. Assigning through `**pp` reaches all the way back to `x`, exactly the same way `*p = 7` did in the pointer lesson — there's just one extra hop to make first.

<CodeBlock lang='c' linesJson='["#include &lt;stdio.h&gt;", "", "int main(void) {", "    int x = 42;", "    int *p = &amp;x;", "    int **pp = &amp;p;", "", "    **pp = 7;", "    printf(\"x = %d\\n\", x);", "    return 0;", "}"]' runnable expectedOutput='x = 7' />

## The real reason to reach for **: changing the pointer itself

So far this just looks like a longer way to change `x`. The real payoff shows up when a function needs to change which address <i>the caller's own pointer</i> holds — not the thing it points to. Passing an `int *` by value only gives the function a copy of the address; changes to that copy never reach the caller. Passing an `int **` gives the function the address of the caller's pointer variable, so it can reassign it directly.

In the lab below, `point_to_new_value` allocates a fresh int on the heap and makes the caller's `p` point at it — `p` starts as `NULL` in `main` and ends up pointing at real, heap-allocated memory once the function returns.

<CodeBlock lang='c' linesJson='["#include &lt;stdio.h&gt;", "#include &lt;stdlib.h&gt;", "", "void point_to_new_value(int **pp, int value) {", "    int *fresh = malloc(sizeof(int));", "    *fresh = value;", "    *pp = fresh;", "}", "", "int main(void) {", "    int *p = NULL;", "    point_to_new_value(&amp;p, 100);", "    printf(\"*p = %d\\n\", *p);", "    free(p);", "    return 0;", "}"]' runnable expectedOutput='*p = 100' />

## Where you'll see this pattern again

Double pointers show up constantly once you know to look. Every C `main` can be written as `int main(int argc, char **argv)` — `argv` is a pointer to the first element of an array of `char *` strings, so it's a double pointer by nature. Functions that need to grow or replace an array they don't own take an `int **` or similar for the same reason `point_to_new_value` did above.

You won't need triple pointers for anything in this level, and if you ever reach for one, it's usually a sign the design needs rethinking. But `int **` is genuinely common, and now you have the two ideas — indirection, and 'a pointer is just a variable' — needed to read it comfortably wherever it turns up, including the structs and dynamic-memory lessons still ahead in this level.
