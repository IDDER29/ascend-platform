Last lesson gave you the map: stack near the top of the address space growing down, heap lower down growing up. Now you get to watch both of them actually work — because "where a variable lives" turns out to be one of the most consequential decisions in a program, and it's a decision you'll make explicitly, in C, starting next level.

The stack and the heap solve the same basic problem — where to put data while a program runs — with two completely different strategies. Understanding both, and when each one is used, is what makes concepts like function calls, recursion, and dynamic memory click instead of feeling arbitrary.

## The stack: a history of function calls

Every time one function calls another, the CPU pushes a <b>stack frame</b> onto the stack: the address to return to once the called function finishes, room for that function's local variables, and often a copy of a few registers it plans to reuse. The stack pointer (SP, from Lesson 1) just moves down to carve out that space — pushing is cheap, just a subtraction.

When the function returns, its frame is popped — the stack pointer moves back up, the return address is read to know where execution resumes, and every local variable that lived in that frame simply ceases to exist. Nothing is zeroed out or cleaned up; the bytes are just no longer claimed by anyone, which is exactly why using a variable's address after its function has returned is a serious bug you'll learn to avoid in Level 02.

## The heap: memory you ask for by name

The stack is great for data that comes and goes in strict order with function calls — but not everything fits that pattern. Sometimes you don't know how much memory you'll need until the program is running, or you need something to outlive the function that created it. That's what the <b>heap</b> is for: a region of memory you request from, in a specific amount, at runtime.

Unlike the stack, heap memory doesn't come and go automatically. You (or, later, your C code calling `malloc`) explicitly ask an allocator for a chunk of a given size, and that chunk stays reserved — even after the function that requested it returns — until it's explicitly given back. There's no automatic order to it either: chunks can be requested and released in any sequence, which is exactly why heap management needs bookkeeping the stack never has to do.

## Choosing stack vs heap

The stack wins on speed and simplicity: allocating a local variable is just moving the stack pointer, and cleanup is automatic and instant. Its cost is limited size (typically a few megabytes total) and a strict lifetime — a stack variable is gone the instant its function returns, no exceptions.

The heap wins on flexibility: it can be enormous (limited mainly by RAM), and its contents live exactly as long as you want them to. Its cost is that <i>you</i> become responsible for that lifetime — forget to release a chunk and you've leaked memory; release it too early, or use it after releasing it, and you've triggered undefined behaviour. You'll spend a good chunk of Level 02 learning to manage that trade-off deliberately.
