A pointer gets you to one address. A `struct` gets you to a whole neighborhood of them &mdash; one name for a bundle of related variables sitting side by side in memory. It's how C models a point, a person, a network packet: anything made of more than one piece of data that belongs together.

A `union` looks almost identical on the page but means something completely different: instead of laying its members side by side, it makes them all share the exact same bytes. One is &ldquo;and,&rdquo; the other is &ldquo;or.&rdquo; Let's build both, and run the code that proves the difference.

## Bundling related data

A `struct` gives a name to a group of variables and lays them out one after another in memory, as a single unit. `struct point { int x; int y; };` defines the shape; `struct point a = { 3, 4 };` actually carves out the memory and fills it in.

The dot operator, `.`, reaches into that block by name: `a.x` and `a.y` are just labels for two neighbouring addresses &mdash; not two unrelated variables that happen to travel together.

## Reaching in through a pointer

Structs and pointers combine constantly &mdash; you'll pass a `struct point *` into a function instead of copying the whole struct every time it's used. Through a pointer, `(*p).x` would technically work, but C gives you a cleaner shorthand: `p->x` means exactly the same thing.

Everything you already know about dereferencing still applies. `p->x` doesn't just read a's field &mdash; writing to it writes a's actual memory, because p is still just an address.

<CodeBlock lang='c' linesJson='["#include &lt;stdio.h&gt;", "", "struct point {", "    int x;", "    int y;", "};", "", "int main(void) {", "    struct point a = { 3, 4 };", "    struct point *p = &amp;a;", "    p-&gt;x = 7;", "    printf(\"x = %d, y = %d\", p-&gt;x, a.y);", "    return 0;", "}"]' runnable expectedOutput='x = 7, y = 4' />

## Padding: why sizeof surprises you

You might expect `sizeof(struct example)` to equal 1 + 4 = 5 for a char followed by an int. On most common 64-bit platforms it comes back 8. The compiler quietly inserts padding bytes so that i starts at an address divisible by 4 &mdash; most CPUs read multi-byte values fastest when they're aligned that way.

This isn't a bug or a waste; it's a deliberate trade of a few bytes of memory for a lot of speed. It does mean you should never assume a struct's size by adding up its members yourself &mdash; always ask `sizeof`.

<CodeBlock lang='c' linesJson='["#include &lt;stdio.h&gt;", "", "struct example {", "    char c;", "    int  i;", "};", "", "int main(void) {", "    printf(\"sizeof(struct example) = %zu\", sizeof(struct example));", "    return 0;", "}"]' runnable expectedOutput='sizeof(struct example) = 8' />

## Unions: one address, many types

A `union` is declared exactly like a struct, but every member starts at the same address and shares the same bytes. Write i and you've also just overwritten whatever bit pattern f was holding &mdash; there's only one slot, not two.

That's why `sizeof` a union equals the size of its <i>largest</i> member, not the sum of all of them: the compiler only needs to set aside enough room for whichever member happens to be in use.

<CodeBlock lang='c' linesJson='["#include &lt;stdio.h&gt;", "", "struct s_pair { int i; float f; };", "union  u_pair { int i; float f; };", "", "int main(void) {", "    printf(\"struct: %zu bytes, union: %zu bytes\",", "           sizeof(struct s_pair), sizeof(union u_pair));", "    return 0;", "}"]' runnable expectedOutput='struct: 8 bytes, union: 4 bytes' />
