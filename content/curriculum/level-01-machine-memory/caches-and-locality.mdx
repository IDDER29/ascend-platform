By now you've got registers (blazing fast, tiny) and memory (huge, and — as promised — much slower). This lesson finally puts a number on "much slower," and introduces the trick that keeps modern CPUs from spending most of their time waiting around: the <b>cache</b>.

This isn't just trivia for hardware nerds. Once you understand why caches work, you'll be able to look at two pieces of code that do the "same" amount of work and correctly predict which one runs dramatically faster — a skill that starts mattering the moment you write your first C loops in Level 02.

## The speed gap nobody warned you about

A register access takes about one CPU clock cycle. A trip out to main RAM can take a couple of hundred cycles — call it roughly a hundred times slower. If the CPU had to wait on RAM for every single value it needed, it would spend the overwhelming majority of its time doing nothing at all, which is a spectacular waste of a chip you paid for.

The fix is a <b>memory hierarchy</b>: a series of storage layers between the CPU and RAM, each one bigger but slower than the last. <b>L1 cache</b> sits closest to the CPU (tiny, almost register-speed); <b>L2</b> and <b>L3 cache</b> sit further out (bigger, a bit slower); then main <b>RAM</b>; then, way out at the bottom, <b>disk</b> — millions of times slower than a register. Most of the time, the CPU finds what it needs in a cache and never has to make the slow trip at all.

## Locality: why caching even works

Caching only works because real programs aren't random — they exhibit <b>locality</b>. <b>Temporal locality</b> means: if a program used a piece of data once, it's likely to use that exact same piece again soon (think of a loop counter, checked every iteration). <b>Spatial locality</b> means: if a program used one address, it's likely to use a <i>nearby</i> address soon too (think of walking through an array element by element).

Caches exploit spatial locality directly: when the CPU asks for one byte that isn't in cache, the cache doesn't fetch just that byte — it fetches an entire <b>cache line</b>, typically 64 contiguous bytes, betting that you'll want your neighbors soon. Get that bet right consistently, and a program can run almost entirely out of cache, barely touching slow RAM at all.

## Cache-friendly vs cache-hostile code

This is why an array and a linked list, holding the exact same data, can perform completely differently. An array's elements sit back-to-back in memory (the same contiguous layout from the heap/stack discussion two lessons ago) — walking through it in order is a spatial-locality dream, and cache lines keep feeding the CPU exactly what it's about to ask for next.

A linked list's nodes, by contrast, can be scattered anywhere the allocator happened to put them. Following each `next` pointer can jump to a completely different, uncached region of memory — a likely <b>cache miss</b> almost every single step, each one costing the full RAM-speed penalty from the pyramid above.
