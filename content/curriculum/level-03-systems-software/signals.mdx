Press Ctrl-C while a program runs and something interrupts it <i>immediately</i>, mid-instruction, with no cooperation required from the program itself. That's a <b>signal</b> — a tiny, asynchronous notification the kernel delivers to a process, completely independent of whatever that process happens to be doing at the time.

Signals are how the outside world reaches into a running process without a syscall being called <i>by</i> that process. Your shell needs this badly: when someone presses Ctrl-C, it has to interrupt the command it's running <i>without dying itself</i> — and that only works if you understand exactly how signals are delivered and caught.

## What a signal actually is

A signal is just a small integer (like `SIGINT` = 2, `SIGKILL` = 9, `SIGTERM` = 15) that the kernel delivers to a process — triggered by a keypress (Ctrl-C), another process calling `kill()`, or the kernel itself noticing something (like a segfault, which raises `SIGSEGV`).

Every signal has a <b>default disposition</b> — usually terminate, sometimes ignore, sometimes stop the process — but a process can override most of them by installing its own <b>handler</b>, a function that runs instead of the default when that signal arrives.

## Installing a handler

`sigaction()` registers a function to run when a given signal arrives, replacing that signal's default behavior. Because a handler can interrupt your program at <i>any</i> instruction, the rules for what's safe to do inside it are strict — the safest pattern is to do almost nothing: just set a flag and let your normal code react to it.

That's why the example below uses `volatile sig_atomic_t` instead of calling `printf()` directly inside the handler — printf isn't guaranteed safe to call from a signal handler, since it might interrupt an in-progress printf call from the main program.

## Some signals can't be caught

`SIGKILL` (9) and `SIGSTOP` (19) can never be caught, blocked, or ignored — the kernel handles them itself, guaranteeing there's always a way to force-kill or force-pause any process, even a buggy one that ignores everything else. That's why `kill -9` always works.

For your shell project, the signal you'll actually care about most is `SIGINT`: your shell process needs its own handler so that Ctrl-C doesn't kill the shell itself, while the child process running the current command is left to receive SIGINT normally (or gets it forwarded) and stop. Get this wrong and every Ctrl-C during a long-running command kills your entire terminal session instead of just the command.
