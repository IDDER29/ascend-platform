You've now watched instructions move values between registers while the ALU works on them. But registers are scarce — a dozen or so slots — and real programs juggle far more data than that. The overflow goes to <b>memory</b>, an enormous array of numbered bytes sitting outside the CPU, and every one of those numbers is an <b>address</b>.

Before you can understand pointers, the stack, or the heap, you need a map. This lesson draws it: what a process's address space actually looks like, region by region, and why it's organized the way it is.

## Addresses you can't see yet

Think of memory as an absurdly long street of houses, each holding exactly one byte, each with a house number — its <b>address</b> — starting at 0 and counting up. An instruction like "store this value" doesn't just store it somewhere vague; the control unit hands the memory system an exact address, and that byte (or several consecutive bytes, for anything bigger than 8 bits) is what gets written.

Here's the twist that matters for everything that follows: your program doesn't see real, physical addresses. The operating system gives every running process its own private, virtual <b>address space</b> — a clean number line from 0 up to some huge maximum, as if the process owned all of memory itself. What we're about to lay out is that private space, not the physical RAM chips underneath it.

## The layout, region by region

Slice a typical process's address space from low addresses to high, and you'll find the same handful of regions in almost every program you'll ever run: <b>text/code</b> at the bottom (the compiled instructions themselves, read-only), then <b>initialized and uninitialized data</b> (global variables), then the <b>heap</b>, then a large gap of unused address space, and finally the <b>stack</b> up near the top, just below space the kernel reserves for itself.

Two of those regions are special because they change size while the program runs: the <b>heap grows upward</b>, toward higher addresses, as the program asks for more dynamic memory; the <b>stack grows downward</b>, toward lower addresses, as functions call other functions. Everything else is fixed the moment the program is loaded.

## Why the layout looks like this

Putting the stack and heap at opposite ends with a gap between them is a deliberate trick: nobody knows in advance exactly how large either one will need to be, so giving each of them room to grow into that shared gap lets both expand as needed, without anyone having to guess a fixed size for either one up front.

It also bakes in a natural failure mode: if a program recurses too deeply or allocates too much dynamic memory, the two growing regions can meet in the middle. That collision is exactly what's behind the phrase <b>stack overflow</b> — a term you'll meet for real once you're writing recursive C functions in Level 02.
