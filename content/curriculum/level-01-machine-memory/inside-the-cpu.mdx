In Level 00 you built the <b>fetch-execute loop</b> up from gates: fetch an instruction, decode it, execute it, repeat — forever, thousands of millions of times a second. That loop is real, but so far it's been an idea. This lesson puts a body on it: the actual pieces of silicon that do the fetching, the deciding, and the arithmetic.

Three characters do almost all the work inside a CPU: a small set of <b>registers</b> for holding values right now, an <b>ALU</b> (arithmetic logic unit) for doing math and comparisons on those values, and a <b>control unit</b> that steers the whole show. Get comfortable with these three, and every instruction you'll ever read — in assembly, in a debugger, in a crash dump — starts making sense.

## From fetch-execute to real hardware

Recall the loop: <b>fetch</b> the next instruction from memory, <b>decode</b> what it's asking for, <b>execute</b> it, and move on. Every step in that loop needs somewhere to put things down while it works — the instruction itself, the numbers it operates on, the result. That "somewhere" is a tiny, blazing-fast set of storage slots built directly into the CPU chip: <b>registers</b>.

The <b>control unit</b> is the part that actually drives the loop — it holds the address of the next instruction, tells the ALU which operation to perform, and tells the register file which slots to read from and write to. Nothing moves inside a CPU without the control unit orchestrating it.

## Registers: the CPU's scratchpad

A <b>register</b> is a storage slot that lives inside the CPU itself, not out in main memory. On a typical modern machine there are only a handful of general-purpose ones — think a dozen or so — each holding a single word (commonly 32 or 64 bits). Because they sit right next to the ALU with no bus, no address decoding, and no waiting, reading or writing a register takes a single clock cycle. Reading from memory, as you'll see in Lesson 5, can take a hundred times longer.

A few registers have jobs so important they get their own names. The `program counter` (PC, sometimes called the instruction pointer) always holds the address of the next instruction to fetch — it's what keeps the loop looping. The `stack pointer` (SP) tracks the top of the call stack, which you'll meet properly in Lesson 4. Everything else — the <i>general-purpose</i> registers — is yours (or the compiler's) to use as scratch space.

## The ALU: where decisions and math happen

The <b>ALU</b> takes one or two values from registers, applies an operation — add, subtract, AND, OR, shift, compare — and produces a result, usually written back into a register. This is the same digital logic from Level 00: adders and multiplexers built from gates, just wired up to accept whichever operation the control unit selects for this instruction.

Every ALU operation also sets a small set of one-bit <b>flags</b>: was the result zero? negative? did it overflow the register's width? did a carry fall out the top? Those flags aren't trivia — they're how the CPU makes decisions. A `compare` is really just a subtraction whose result is thrown away and whose flags are kept, and next lesson's conditional jumps read those flags to decide where to go next.
