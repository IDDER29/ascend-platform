In Level 01 you learned that memory is really just an enormous strip of numbered boxes — bytes, each with its own address, sitting quietly whether anyone's using them or not. Now it's time to make that idea do real work: every variable you write in C claims some of those boxes, and the type you give it decides exactly how many.

This lesson is the bridge. By the end you'll look at a line like `int n = 42;` and see past the syntax — four specific bytes at a specific address, holding a specific pattern of bits. That's the ground floor everything else in this level is built on, including the pointer you'll meet two lessons from now.

## Bits become bytes become variables

Every value in a C program is bits underneath, and Level 01 already showed you those bits are grouped into 8-bit bytes, each with its own address. C's job is to let you name a chunk of that grid instead of poking at raw addresses by hand. Write `int n = 42;` and the compiler picks a free address, reserves enough bytes for an `int`, and stores the bit pattern for 42 there.

How many bytes it reserves depends entirely on the type. A `char` needs exactly 1 byte because that's how big a byte is defined to be. An `int` is usually 4 bytes and a `double` is usually 8 — <b>usually</b>, because C guarantees minimums, not exact sizes, which is exactly why you ask the compiler with `sizeof` instead of assuming.

## A program's memory map

Zoom out and every running C program divides its address space into a few regions with different jobs. Compiled instructions and constants sit in one region; global variables that exist for the program's whole life sit in another. Two more regions matter most day to day: the `stack`, where local variables and function calls live, and the `heap`, where memory you explicitly ask for lives. You'll spend all of the next lesson on that pair.

The stack and heap don't just sit side by side — on most systems they start at opposite ends of the available space and grow toward each other. The stack grows downward as functions call other functions; the heap grows upward as you allocate more memory. Neither is infinite, which is exactly why 'stack overflow' and 'out of memory' are both real, distinct failures.

## Types are the compiler's promise, not the machine's

Here's the twist worth sitting with: the memory itself doesn't know anything about `int` or `double` — it's just bytes. Types are a promise the compiler keeps on your behalf, tracking how many bytes belong to each variable and how to interpret the bits it finds there. That's also why two variables declared back-to-back usually land near each other, but never take that for granted — the compiler is free to reorder or pad things however it likes.

Keep that picture — bytes, addresses, and a compiler that remembers what each address means — in your head. It's exactly what you need for the next lesson's tour of the stack and heap, and two lessons from now, for the idea that a pointer is nothing more than a variable that stores one of these addresses.
