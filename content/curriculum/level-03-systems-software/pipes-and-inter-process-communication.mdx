Processes are isolated from each other by design — one process can't just read another's variables. So how does `ls | grep .c` work, where the output of one program becomes the input of another? The kernel builds a small, temporary conduit between them called a <b>pipe</b> — and once you understand it, that `|` character stops being shell syntax and becomes three lines of C you fully understand.

This is the lesson where fork() from last time and syscalls from the lesson before that combine into something genuinely useful: two separate processes, talking to each other, safely, through the kernel.

## A pipe is a kernel-managed queue

`pipe(fds)` asks the kernel for a small in-memory buffer and hands you two file descriptors that point at opposite ends of it: `fds[0]` for reading, `fds[1]` for writing. Whatever bytes you write in one end come out the other, in order, like a queue.

It's <b>unidirectional</b> — you can't write to fds[0] or read from fds[1] — and it's <b>bounded</b>: the kernel buffer holds a fixed amount (commonly 64KB on Linux). Write past that and your process blocks until a reader drains some of it; read from an empty pipe and you block until a writer adds some.

## Wiring two processes together

A pipe is only useful once two <i>different</i> processes hold its ends — and `fork()` is how you get that: create the pipe first, then fork, and the child inherits copies of both fds. Each process then closes the end it doesn't need — the reader closes the write end, the writer closes the read end.

That closing step isn't just tidy; it's what lets `read()` ever return 0 (end-of-file). As long as <i>any</i> process still has the write end open, a reader waiting on an empty pipe will block forever instead of seeing EOF.

## How your shell will wire ls | grep .c together

A real shell pipeline is just this pattern scaled to two <i>fresh</i> processes running different programs, connected instead of hand-written printf/read code. The shell calls `pipe()` once, forks twice, and in each child redirects a standard stream onto the pipe before calling `execve()` — the redirection trick itself is coming up in a later lesson.

Notice the parent (your shell) never touches the data at all — it just builds the plumbing and gets out of the way. That's exactly the shape you'll write inside Shellforge.
