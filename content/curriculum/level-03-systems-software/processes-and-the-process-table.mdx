Every program you run — your editor, your browser, that C binary you just compiled — is a <b>process</b> to the operating system: a live, running instance with its own memory, its own open files, and one all-important number, its `PID`. The kernel doesn't run your code as a favor — it tracks every single process in a table and decides, thousands of times a second, whose turn it is to use the CPU.

This lesson is where "the shell" stops being magic. Type `ls` at a prompt and, behind the scenes, a brand-new row appears in that process table, runs for a few milliseconds, and vanishes. By the end you'll know exactly how that row gets created, what's inside it, and what happens if nobody cleans up after it.

## What lives in a process table entry

The kernel keeps a table with one row per running process — think of it as the OS's own spreadsheet of everything alive on the machine right now. Each row holds a <b>PID</b> (its ID), a <b>PPID</b> (its parent's ID), a <b>state</b> (running, sleeping, stopped, or zombie), and pointers to the process's memory and its table of open files.

You never edit this table directly — you only ever cause the kernel to add or update rows in it, by calling functions like `fork()`, `exit()`, and `wait()`. Every process except the very first has a parent, which is why the table naturally forms a tree with PID 1 (`init` or `systemd`) at the root.

## Cloning yourself: fork()

`fork()` is the one syscall that returns <i>twice</i>. It clones the calling process — same code, same variables, same open files — into a brand-new process table entry, and then both the original (parent) and the clone (child) continue running from the exact same line, right after the call.

The only thing that tells them apart is the return value: the parent gets back the child's PID (a positive number), the child gets back `0`, and a failed fork returns a negative number to whoever called it (there's only one "whoever" then, since the child never got created). That single `if (pid == 0)` check is how every fork-based program branches into "parent work" and "child work."

## Reaping children: wait() and zombies

When a child calls `exit()`, it doesn't vanish immediately — its process table entry sticks around as a <b>zombie</b>, just long enough to hold its exit status for the parent to collect. If the parent never asks, that zombie entry lingers until the parent itself exits or explicitly reaps it.

`waitpid()` is how a parent collects that status and frees the entry. This matters enormously for the shell you're about to build: every command you run is a child, and your shell must reap it or your terminal will slowly accumulate zombies every time you run a command.
